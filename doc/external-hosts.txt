# External Host Workflow

A web app is registered normally first, using its platform host name. Any number of external hosts
are attached later to an existing deployment.

Registration happens in a standard way, for example:

    register-web w1 welcome.examples.org certbot

the platform creates the regular deployment host:

    w1.xqiz.it

No external host is assigned during registration.

# External Host Setup

After registration, the customer can attach an external host, for example:

    add-external w1 welcome.acme.com

The platform:

1. Stores the external host in WebAppInfo:

       "welcome.acme.com"

2. Creates a route and a self-certified keystore entry for the external host.

# DNS Setup

After external-host setup, the customer points the external host at the generated unique route. For
example:

    welcome -> 23940954.welcome.acme.com.xqiz.it

The "xqiz.it" part is neccessary for routing; the "welcome.acme.com" part serves as a documentation
for the customer; the UUID part is used for the platform verification purposes.

Certificate Renewal

After DNS is configured, the customer renews the certificate for the deployment. During renewal, the
platform:

1. Calls ensureCertificate().
2. Sees the existing self-signed placeholder for WebAppInfo.externalHost.
3. Checks the DNS record ensuring it confirms to the required shape (UUID at all).
4. Uses the configured provider to replace that placeholder with a provider-issued certificate.

If certificate issuance fails, the stored WebAppInfo does not change and the self-signed
placeholder remains in the keystore. A later renewal can retry the provider-issued certificate.
